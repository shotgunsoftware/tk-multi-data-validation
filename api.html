API Reference¶

The Data Validation API provides an interface to use components of the App, without running the App from a GUI.

Validation Manager¶

class tk_multi_data_validation.api.ValidationManager(rule_settings=None, include_rules=None, exclude_rules=None, validation_logger=None)[source]¶

Bases: object

Manager class for data validation.

This class is responsible for retrieving the data validation rules from the current bundle’s settings, and creating the set of ValidationRule objects that define how the data should be validated.

This class object may be passed to the tk_multi_data_validation.widgets.ValidationWidget to help manager validation rules, and executing their actions.

It also coordinates the execution of validation rule check and fix functions.

Initialize the validation manager from the settings data.

Parameters:
  • rule_settings (dict) – The rule settings to use for this manager. Default is to use the current bunlde’s settings.
  • include_rules (list<str>) – List of rule ids to include from the app’s default rules list.
  • exclude_rules (list<str>) – List of rule ids to exclude from the app’s default rules list.
  • validation_logger – This is a standard python logger to use during validation. A default logger will be provided if not supplied.

resolve(pre_validate=False, post_validate=False)[source]¶

Resolve the current data violations found by the validate method.

The fix function executed to resolve the DCC data violations is implemented by the ValidationRule (e.g. the manager does nothing to validate the data, it is just responsible for executing the fix functions).

Parameters:
  • pre_validate (bool) – True will run the validation step before the resovle step, to ensure the list of items to resolve is the most up to date.
  • post_validate (bool) – True will run the validation step after the resolve step, to check that the scene data is valid after resolution steps applied.

resolve_rules(rules, fetch_dependencies=None, emit_signals=True)[source]¶

Resolve the given list of rules.

Steps to resolve rules 1. Iterate over all rules

  1. If the rule has no dependencies - resolve it immediately
  2. If the rule has dependencies - add it to the queue to process later

  1. If fetch_dependencies is not explicitly set as False, check if all dependencies are provided

    1. If missing dependencies and fetch_dependencies not explicitly set to True then prompt user to fetch and resolve dependencies
    2. If fetch_dependencies is explicitly set to True or user answered YES to (a), then try to find any missing dependencies in the manager, and process them as done with the other rules

  2. Process the queue of rules (that have dependencies)

    1. If the rule’s dependencies have been resolved or ignored - now resolve it and mark it as resolved
    2. If the rule’s dependencies have not been resolved - add it back to the end of the queue

Time complexity (n = number of rules)

  1. O(n)
  2. O(n^2) - worst case

NOTE: Time complexity is not the best with O(n^2) but is probably good enough since the list of validation rules is not expected to be so large. If the number of rules does grow to be a large list, then this fix operation may need to be optimized (e.g. build a dependency tree that defines the order of fixing the rules).

Parameters:
  • rules (list<ValidationRule>) – The list of rules to resolve.
  • fetch_dependencies (bool) – Set to True to ensure all dependencies for a reule are resolved before the rule itself is resolved. Set to False will not fetch any missing dependencies to resolve.
  • emit_signals (bool) – Set to True to emit notifier signals for resolve operation.
